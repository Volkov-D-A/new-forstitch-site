# Подготовка макета к подключению бэкенда

## Текущая структура

- `src/main.tsx` — подключает `BrowserRouter`, глобальные стили и корневой `App`.
- `src/App.tsx` — корневой контейнер: routes, корзина и toast.
- `src/pages/` — отдельные страницы главной, каталога, товара, галереи, блога и инструкции покупки.
- `src/components/` — переиспользуемые UI-компоненты: `Header`, `Footer`, `ProductCard`, `CartDrawer`, `SImg`, `Toast`, `Stitch`, `EmptyState`.
- `src/components/home/` — секции и hero-варианты главной страницы.
- `src/styles/` — глобальные стили и стили главной.
- `src/mocks/` — временные mock-данные, разделённые по доменам: `products`, `categories`, `blog`, `gallery`, `siteContent`.
- `src/types/site.ts` — доменные типы данных для API и UI.
- `src/utils/routes.ts` — единые URL-хелперы для категорий, товаров и статичных разделов.
- `src/utils/format.ts` — небольшие форматтеры, сейчас pluralization.
- `src/services/siteApi.ts` — точка подключения API. Если `VITE_API_BASE_URL` не задан, используются mock-данные.
- `src/hooks/useSiteData.ts` — загрузка данных сайта, состояние loading/error.

## Роутинг

- `/` — главная.
- `/shop` — весь каталог.
- `/shop/:categoryId` — каталог конкретной категории.
- `/product/:productId` — карточка товара.
- `/gallery` — галерея.
- `/blog` — блог.
- `/how-to-buy` — как купить.
- `/howto` — legacy redirect на `/how-to-buy`.

## Страницы

- Главная: использует новинки, автора, отзывы, галерею и hero-товар.
- Магазин: список товаров, фильтр по категории, сортировка на клиенте.
- Карточка товара: данные товара, категория, связанные товары.
- Галерея: список изображений отшивов и lightbox.
- Блог: список публикаций.
- Как купить: шаги покупки, CTA и подборка стартовых схем.

## Переиспользуемые компоненты

- `ProductCard` подходит для главной, каталога, похожих товаров и подборок.
- `SImg` централизует отложенную/очередную загрузку изображений.
- `Header` и `Footer` строят меню из категорий.
- `CartDrawer` пока клиентский, но готов принимать товары и форматтер цены извне.

## Данные с бэкенда

Фронт уже готов к доменным endpoint-методам:

```http
GET /categories
GET /products
GET /products/:productId
GET /gallery
GET /blog
```

Текущий `getSiteData()` остаётся агрегатором для UI и собирает форму:

```js
{
  categories: [],
  products: [],
  testimonials: [],
  gallery: [],
  author: {},
  blog: [],
  howToBuy: []
}
```

Позже можно добавить:

- `GET /products?category=&sort=`
- `GET /blog/:slug`
- `GET /site-content`
- `POST /cart` или `POST /orders`
- `POST /subscribe`

## Динамические данные

- Категории каталога.
- Товары: цена, статус новинки, изображения, характеристики, наличие/публикация.
- Галерея отшивов.
- Блог и новости.
- Отзывы, если планируется управлять ими из админки.
- Авторский блок, если будет CMS.
- Корзина, заказ, оплата, подписка.

## Статичные данные

- Базовая навигация: главная, магазин, галерея, блог, как купить.
- Тексты кнопок и системные UI-сообщения.
- Шаги `howToBuy` можно оставить статикой, если процесс покупки не редактируется из админки.
- Footer-копирайт и внешние ссылки можно оставить статикой либо вынести в `site-content`.

## Что важно для бэкенда

- У товаров нужен стабильный `id` или `slug`; сейчас навигация использует `id`.
- Категория товара сейчас хранится как `cat`, значение должно совпадать с `categories[].id`.
- Цена приходит числом в рублях, форматирование делает фронт.
- Изображения ожидаются абсолютными URL.
- Для товаров без изображения фронт показывает placeholder.
- Для полноценного магазина корзину лучше переводить с массива `productId` на позиции `{ productId, quantity }`.

## Как подключить API

Создать `.env` в `frontend`:

```bash
VITE_API_BASE_URL=https://example.com/api
```

После этого `src/services/siteApi.js` начнет запрашивать `GET https://example.com/api/site`.
